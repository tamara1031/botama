# ADR 0001: 実装言語として Go を採用

- **ステータス**: Accepted
- **決定日**: 2026-05-05

## コンテキスト

Discord bot のランタイム言語を選定する。主な候補は以下の通り。

| 言語 | 利点 | 課題 |
|---|---|---|
| Go | 単一静的バイナリ、軽量 Docker イメージ（distroless/scratch）、標準ライブラリが充実、インターフェースによるモジュール分離が容易 | discord.js ほどエコシステムが成熟していない |
| TypeScript (Node.js) | discord.js が成熟、スラッシュコマンド・コンポーネントの実装例が豊富 | Node.js ランタイム込みのイメージが重い（200 MB+）、型安全だが実行時オーバーヘッドあり |
| Python | 開発速度が速い | ランタイムが重い、GIL によるスレッドモデルの制限 |

## 決定

**Go** を採用する。

## 根拠

1. **軽量イメージ**: CGO 無効の静的バイナリとして distroless/scratch ベースイメージに乗せると最終イメージが約 10–20 MB になり、K8s での高速 pull が可能。
2. **K8s 親和性**: SIGTERM ハンドリング・非 root 実行・読み取り専用ファイルシステムとの相性が良い。
3. **モジュール分離**: Go の `interface` は追加メソッドなしでプラグインパターンを実現でき、registry 経由での有効化・無効化が型安全に書ける。
4. **標準ライブラリ**: `log/slog`（構造化ログ）、`os/signal`（SIGTERM）が追加依存なしで利用可能。
5. **discordgo**: `github.com/bwmarrin/discordgo` は安定しており、メッセージイベント・スラッシュコマンド両方に対応。

## トレードオフ

- スラッシュコマンドの実装は discord.js より若干冗長になるが、現時点でモジュールはメッセージベースのコマンドから始めるため影響なし。
- Go に不慣れな場合の学習コストがあるが、モジュール追加は `Module` インターフェースを実装するだけであり、参入障壁は低い。
